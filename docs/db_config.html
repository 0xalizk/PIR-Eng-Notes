a XPIR (2016) is a system paper with auto-parameter optimization. Benchmarks span many DB sizes via an automated search over recursion depths and ring parameters — no single fixed configs to list. Values read from Figures 6–8 (log-scale).

b Addra/FastPIR has two distinct evaluation contexts: PIR microbenchmarks (Table 1: 32B entries at various N) and the Addra voice-call system benchmark (Section 6.5: 32,768 mailboxes × 96B messages). reported.json uses the system benchmark (32768 × 96B). Values are Tier 2 (approximate from figures).

c CwPIR's benchmark configs (220–228 × 288B) match SealPIR's configs because CwPIR is evaluated as a drop-in replacement for SealPIR. Entry size is BFV plaintext payload; CwPIR is keyword PIR — the "288B" is the record payload, not the identifier.

d FrodoPIR tests a grid of 5 N values (216 to 220) × 2 entry sizes (1 KB, 8 KB) = 10 configurations total. All in Table 6.

e ThorPIR has concrete parameter choices in Table 2 (N=230, 360B entries, 1024-query budget) but no running implementation. Server time derived from end-to-end time / Q queries. The parameters are the paper's own specification, not our estimate — hence "Analytical" rather than "Estimated."

f HintlessPIR fixes N=220 and scales entry size (288B to 8KB to 2MB). The 2MB entry config produces ~560 GB total DB — the largest tested by any scheme — but with only 220 ≈ 1M entries.

g YPIR's headline benchmark is "32 GB, 1-bit records" (238 entries × 1 bit), which is its primary throughput result. The 256B-entry configs (216–224) may be from comparison evaluation with other schemes. Needs verification whether these are YPIR's own runs or competitor data.

h WhisPIR values are Tier 2 — read from log-scale figures, not exact tables. Paper also tests up to 32 GB in Figure 4, but specific (N, entry_size) pairs are not clearly separable from the charts.

i SimplePIR and DoublePIR keep total DB size constant at ~1 GB and vary the entry size / entry count tradeoff. All four configs represent the same ~1 GB database accessed differently.

j VeriSimplePIR evaluates a scaling benchmark from 4 GiB to 256 GiB. Figures 7–8 show performance as a function of DB size on log-scale axes. Individual (N, entry_size) pairs not extractable — the paper plots continuous scaling curves, not discrete configurations.

k BarelyDoublyEfficient is a pure theory/feasibility result. "1 GB" was chosen for asymptotic estimation in reported.json. The paper proves existence via Williams' technique but the poly(lambda) constants are unknown and potentially large. No concrete parameters specified in the paper.

l IncrementalPIR evaluates iSimplePIR across a wide range (210 to 230 entries, 1-bit to 32-bit entries). The primary benchmark is 1 GB with 1-bit entries (matching SimplePIR baseline). The paper also evaluates a 4 GB password-breach detection scenario.

m CK20, IshaiShiWichs, Plinko, and WangRen have no implementations. "N=220 × 32B" was chosen by the notes author to match Group D conventions and produce comparable asymptotic estimates. The papers themselves only state bounds in terms of N, not specific benchmark configurations.

n KeywordPIR is a multi-paradigm comparison paper (SealPIR vs. MulPIR vs. Gentry-Ramzan). The 288B config evaluates index PIR; the 40KB config evaluates keyword PIR on a labeled database (100K records).

o DistributionalPIR uses real-world datasets (Twitter). Entry sizes vary per user (tweets). The "~560B" for the 38 GB config is the average tweet size. The smaller configs (5K, 10K entries) are synthetic evaluation points.

Server Performance

A note on the throughput metric in PIR literature

“Throughput” in PIR literature means DB scan rate (GB/s = DB size ÷ server time) — not queries/sec. For schemes with O(N) server work (Groups 1a, 1b, some 2a), this is bounded by memory bandwidth. Preprocessing schemes (Groups 2a, 2b) with sublinear online work can have very low server times but the rate metric is less meaningful.

Evidence from each scheme’s paper that throughput = DB size ÷ server time:

SchemeEvidence
SealPIRTable 2: throughput = DB size ÷ reply generation time
MulPIRTable 4: throughput = DB size ÷ reply generation time (same methodology as SealPIR)
FastPIR (Addra)Table 2: throughput = DB size ÷ server response time
OnionPIRTable 1: throughput = DB size ÷ total server time

SimplePIRTable 1 & Figure 5: throughput = DB size ÷ server time; note near-bandwidth performance
DoublePIRTable 1 & Figure 5: throughput = DB size ÷ server time
CWPIR (FastPIR)Table 1: throughput = DB size ÷ total time
OnionPIR v2Table 2: throughput = DB size ÷ server time

YPIRTable 1: throughput = DB size ÷ server time
RespireTable 2: throughput = DB size ÷ server time
WhisPIRTable 1 & Figure 3: throughput = DB size ÷ server time

Exception — XPIR (2016): XPIR defines “PIR processing throughput” as the rate of homomorphic absorption operations (bits/s through crypto ops), and “user-perceived throughput” as end-to-end including network transfer. Neither is simply DB size ÷ server time. XPIR throughput values in our charts use the standard definition for comparability.

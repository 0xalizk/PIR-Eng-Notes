[\[^13\]](../../../../Schemes/Group%20D%20-%20Client%20Dependent%20Preprocessing/ishaishiwichs_2025/IshaiShiWichs_2025_notes.md#user-content-fn-13-7ba4188844dd97dd8d4df544bd50fe63) — Notes say "Section 4.2 (p.15)." The planar set details (size n^{1/3}, n^{1/6} per dimension) are discussed on p.15 under "Additional details of the preprocessing algorithm," which is technically within Section 4.2. However, the claim that "each hint has n^{1/6} planar sets per dimension" is slightly imprecise: the PDF states each dimension contributes |X_i| (or |Y_i| or |Z_i|) planar sets, and E[|X_i|] = n^{1/6}. So "n^{1/6}" is the expected count, not a fixed count. Minor precision issue.

[\[^14\]](../../../../Schemes/Group%20D%20-%20Client%20Dependent%20Preprocessing/ishaishiwichs_2025/IshaiShiWichs_2025_notes.md#user-content-fn-14-7ba4188844dd97dd8d4df544bd50fe63) — Notes say "Section 3.1 (p.10-11)" and describe "the generic construction piggybacks the next phase's preprocessing over the current phase's queries." The PDF on p.11 describes the generic construction: "the preprocessing is executed upfront, and during each phase of t queries, the client piggybacks the work of the next phase's preprocessing over this phase's queries." The footnote says the hint table is {msg_i, st_i, res_i}. The PDF uses the notation {msg_i, st_i, res_i}_{i in [t]}. Substantively correct, but the footnote says "with each phase covering t queries" -- the PDF says "each phase of t queries" is correct, however the footnote elides that Phase 1 has no piggybacking (preprocessing is executed upfront first). Minor omission.

[\[^19\]](../../../../Schemes/Group%20D%20-%20Client%20Dependent%20Preprocessing/ishaishiwichs_2025/IshaiShiWichs_2025_notes.md#user-content-fn-19-7ba4188844dd97dd8d4df544bd50fe63) — Notes say "Section 4.5 (p.17-19)" and give the bound "1/e^{20} + (1 - 1/e^{20}) * 0.5 <= 0.6." The PDF on p.19 gives exactly: "Pr[failure of finding a good match] <= 1/e^{20} + (1 - 1/e^{20}) * 0.5 <= 0.6." The footnote additionally states "using the birthday bound on hint containment and Markov inequality on the number of good hints." The PDF uses the term "Markov Inequality" (p.19) but does not use the phrase "birthday bound" -- it is a standard coupon-collector / independent-trial calculation. "Birthday bound" is a reasonable colloquial description but not the PDF's own terminology.

[\[^26\]](../../../../Schemes/Group%20D%20-%20Client%20Dependent%20Preprocessing/ishaishiwichs_2025/IshaiShiWichs_2025_notes.md#user-content-fn-26-7ba4188844dd97dd8d4df544bd50fe63) — Notes say "Section 4.2 (p.15)" and describe batching "in batches of n^{2/3} bits." The PDF (p.15) says "for each batch, the client downloads the next n^{2/3} bits from the server." The footnote then says "at any time the client only stores the current batch plus O(n^{2/3}) set descriptions." The PDF says the total client space is O(n^{2/3}) counting both the batch and the set descriptions. The phrasing "current batch plus O(n^{2/3}) set descriptions" could be read as O(n^{2/3}) + O(n^{2/3}) = O(n^{2/3}), which is technically fine but slightly awkward in phrasing compared to the PDF's unified O(n^{2/3}) bound.

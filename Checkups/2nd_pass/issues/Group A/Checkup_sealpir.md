### MINOR Issues

- [\[^2\]](../../../../Schemes/Group%20A%20-%20FHE%20Based%20PIR/sealpir_2018/sealpir_2018_notes.md#user-content-fn-2-e88a300f49f79b7b67cb031ed32aaa68) — The formula in the notes rewrites the PDF's communication complexity in a confusing way. The PDF (Section 3.5, p. 5) states the XPIR complexity as O(Nd * d-root(n)) and SealPIR as O(Nd * ceil(d-root(n)/N)), where Nd is ciphertext size. The notes render this as "O(N * d * N^{1/d})" and "O(N * d * ceil(sqrt_d(n)/N))" which conflates the ciphertext-size variable Nd with the ring dimension N and the number of dimensions d. The intent is recognizable but the notation is garbled.

- [\[^26\]](../../../../Schemes/Group%20A%20-%20FHE%20Based%20PIR/sealpir_2018/sealpir_2018_notes.md#user-content-fn-26-e88a300f49f79b7b67cb031ed32aaa68) — The concrete values in the Complexity table's "Concrete" column silently mix two different database sizes. "Expand 0.11 s" and "Answer 2.01 s" are from n = 1,048,576, while "Answer 0.5 s", "Extract 1.39 ms", and "total 1.65 s" are from n = 262,144 (1.04 + 0.11 + 0.5 = 1.65 s). All individual numbers are accurate per Figure 9, but the table header says "N=2^20" (i.e., n = 1,048,576) while several entries actually come from n = 262,144. The footnote text itself correctly lists both database sizes.

- [\[^40\]](../../../../Schemes/Group%20A%20-%20FHE%20Based%20PIR/sealpir_2018/sealpir_2018_notes.md#user-content-fn-40-e88a300f49f79b7b67cb031ed32aaa68) — The notes present an abbreviated paraphrase as if it were a direct quote: "For even larger databases, since we must use a higher dimension parameter d... the difference becomes less prominent." The actual PDF text (Section 7.2.1, p. 11) reads: "we must use a higher dimension parameter d (Section 3.5), the difference in the number of plaintexts between XPIR and SealPIR (for the same d) becomes less prominent until n is large enough that the second operand in Equation 1 approaches log(t) again." The paraphrase omits the qualifying tail clause.
